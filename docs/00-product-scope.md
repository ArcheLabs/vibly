# Vibly Frontend Preview - Product Scope

## 1. 文档目的

本文档定义 **Vibly PC 端前端预览版** 的范围、边界与实现约束，供 Codex 或前端开发直接据此生成静态预览项目。

本轮目标不是完成正式产品实现，而是完成一套：

- 可运行的 PC 端前端预览
- 结构清晰的 IM 三栏布局
- 覆盖核心页面与关键交互
- 使用本地 mock data 演示产品主流程
- 为后续接入 API、链上状态与正式设计系统预留清晰骨架

---

## 2. 产品定位

Vibly 是一个面向 AI 时代的社交产品。用户既可以以真人身份交流，也可以通过自己拥有或控制的智能体进行交流。产品在交互形态上优先贴近 PC 端 IM，以降低用户理解成本。

本次前端预览版只验证以下问题：

1. 三栏结构是否清晰
2. 聊天主流程是否成立
3. 真人 / 智能体双身份切换是否容易理解
4. 智能体、联系人、发现、钱包、我的等页面是否能组成完整产品骨架

---

## 3. 目标平台

本次仅设计并生成 **PC 端布局**，兼容以下使用场景：

- 桌面浏览器网页端
- 未来 PC 客户端壳（如 Tauri / Electron）

### 布局要求

- 默认面向宽屏桌面端
- 基础可适配较窄桌面宽度
- 不要求本轮完成移动端适配
- 不要求本轮完成响应式断点精细优化

---

## 4. 本轮必须实现的内容

Codex 生成的前端预览版必须包含以下内容。

### 4.1 全局框架

必须包含：

- 左侧导航栏
- 左侧列表区
- 右侧交互区
- 全局页面切换能力
- 统一的 mock 数据来源

### 4.2 一级页面

必须包含以下一级页面：

1. 聊天
2. 智能体
3. 发现
4. 联系人
5. 钱包
6. 我的

### 4.3 公共页面 / 公共详情视图

必须包含：

- 用户信息页
- 智能体信息页

说明：
- 可实现为独立详情页
- 也可实现为右侧详情面板 / 抽屉
- 但交互上必须可被打开和关闭

### 4.4 聊天页必须完整展示的部分

聊天页为本轮优先级最高页面，必须具备：

- 会话列表
- 会话选中态
- 未选中空白态
- 聊天头部
- 聊天记录区
- 输入区
- 身份切换弹窗
- 至少 2 种系统限制或状态演示

### 4.5 必须具备的演示交互

允许使用本地 UI state。必须支持：

- 左侧导航切页
- 点击会话切换当前聊天
- 点击身份切换按钮展开 / 关闭弹窗
- 点击发现页智能体后查看详情
- 从发现页发起聊天并跳转到聊天页
- 点击头像打开用户信息或智能体信息
- 各页面空状态 / 开发中状态展示

---

## 5. 本轮明确不做的内容

Codex 生成时不得擅自实现以下内容。

### 5.1 不接入后端

- 不接入 API
- 不接入 WebSocket
- 不接入数据库
- 不实现消息同步
- 不实现真实数据持久化

### 5.2 不接入链上能力

- 不连接钱包
- 不读取链上账户
- 不发起链上交易
- 不实现智能体真实注册流程
- 不实现收费结算逻辑

### 5.3 不实现真实账户体系

- 不实现登录
- 不实现注册
- 不实现鉴权
- 不实现会话恢复
- 不实现真实权限校验

### 5.4 不实现复杂工程能力

- 不接入全局状态库
- 不接入路由级异步数据加载
- 不实现复杂表单校验
- 不实现国际化
- 不实现主题系统
- 不实现埋点

### 5.5 不实现复杂产品能力

- 不做群聊
- 不做语音视频
- 不做消息撤回 / 编辑
- 不做插件真实运行
- 不做复杂钱包资产管理
- 不做通知中心

---

## 6. 预览版允许使用的能力

以下能力允许使用，且建议使用。

### 6.1 本地 mock data

必须使用本地 mock 数据来支撑页面展示。

### 6.2 局部 UI state

允许使用组件内或页面内的本地状态来演示交互，例如：

- 当前一级页面
- 当前选中的会话
- 当前打开的详情面板
- 当前弹窗是否展开
- 当前用户选中的发言身份
- 当前标签页

### 6.3 本地模拟发送

聊天输入框允许做最简单的本地演示：

- 输入文本
- 点击发送
- 将文本 append 到当前 mock 消息列表

但这仅是 UI 演示，不代表真实消息系统。

### 6.4 占位状态

允许在未实现区域使用占位文案，例如：

- 产品经理正在思考中
- 即将开放
- 暂无数据
- 暂未配置

---

## 7. 技术实现约束

本项目应优先生成一个易于预览和继续开发的前端骨架。

### 7.1 基本要求

- 结构清晰
- 组件尽量复用
- 代码文件命名明确
- mock 数据集中管理
- 不要为预览版引入过重抽象

### 7.2 UI 风格要求

本轮为低保真到中低保真预览，不追求精细视觉稿。

要求：

- 布局清楚
- 区块层级明确
- 功能状态可识别
- 不使用复杂视觉装饰掩盖结构问题

### 7.3 工程目标

第一优先级：跑通主流程预览
第二优先级：代码可维护
第三优先级：为后续正式实现保留扩展空间

---

## 8. 页面优先级

Codex 生成时按以下优先级执行：

### P0

1. 全局 AppShell
2. 聊天页
3. 用户信息页
4. 智能体信息页

### P1

1. 智能体页
2. 发现页
3. 联系人页

### P2

1. 钱包页
2. 我的页

说明：
若实现资源有限，应先保证 P0 和 P1 页面结构正确，再补充 P2。

---

## 9. 输出结果要求

Codex 生成的结果应满足：

1. 本地可运行
2. 默认进入聊天页
3. 可切换所有一级页面
4. 所有核心页面都有基本可视化内容
5. 重要详情页可打开
6. 至少有一条从发现页进入聊天页的完整演示路径

---

## 10. 验收标准

若满足以下条件，则视为本轮前端预览达标：

1. 用户进入后能立即理解这是一个 IM 式桌面产品
2. 用户可明确区分真人与智能体
3. 用户可看懂聊天对象与用户当前身份切换的关系
4. 用户可从多个入口打开用户资料与智能体资料
5. 页面间关系清晰，不依赖解释即可初步演示
6. 整套代码可继续替换 mock 数据为正式实现

---

## 11. 对 Codex 的执行指令建议

可将以下文字直接作为 Codex 任务说明的一部分：

```text
根据 docs/frontend-preview/ 下的文档，生成一个 Vibly PC 端前端预览版。

要求：
1. 只实现前端静态预览，不接入任何 API、链上能力或后端状态。
2. 允许使用本地 mock data 和局部 UI state 来演示交互。
3. 优先完成聊天页，再完成智能体、发现、联系人、钱包、我的。
4. 必须包含用户信息页和智能体信息页的可打开详情视图。
5. 保持低保真到中低保真风格，重点是结构、布局、交互清晰。
6. 尽量复用组件，避免页面间重复实现。
7. 不要擅自扩展群聊、语音视频、插件真实执行、钱包连接等本轮未定义能力。
```
